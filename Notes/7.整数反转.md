# 7.整数反转

## 题目类型

- 数学
- 取模
- 边界判断

## 我的第一反应

- 这题一开始很容易想到转成字符串再反转。
- 字符串写法表面上直观，但会把注意力带到负号、解析、异常这些细节上。
- 这题真正更核心的做法其实是“每次取最后一位，再拼回去”。

## 我这次写法里暴露的问题

- 先走了字符串解法，而不是题目更想考的数学解法。
- 用 `TryParse` 虽然能兜住一部分溢出，但本质上是在“最后交给解析器判断”，不是自己提前判断。
- 负数处理容易出细节问题。
  - 如果直接反转 `x.ToString()`，会把负号也一起反过去。
  - 后来改成先补 `-` 再反转数字，这样方向是对的。
- `x = -x` 这个写法对大多数负数能用，但对 `int.MinValue` 有风险。
  - 因为 `-2147483648` 的相反数超出了 `int` 正数范围。

## 最优解思路

- 不用字符串。
- 每次用 `% 10` 取出最后一位数字。
- 每次用 `/ 10` 去掉最后一位数字。
- 用 `result = result * 10 + digit` 把反转后的数字一点一点拼出来。
- 在真正拼接前，先判断这一步会不会溢出。

## 标准写法

```csharp
public class Solution
{
    public int Reverse(int x)
    {
        int result = 0;

        while (x != 0)
        {
            int digit = x % 10;
            x /= 10;

            if (result > int.MaxValue / 10 || (result == int.MaxValue / 10 && digit > 7))
            {
                return 0;
            }

            if (result < int.MinValue / 10 || (result == int.MinValue / 10 && digit < -8))
            {
                return 0;
            }

            result = result * 10 + digit;
        }

        return result;
    }
}
```

## 每一步在干什么

```csharp
int result = 0;
```

- `result` 用来保存已经反转好的部分。

```csharp
while (x != 0)
```

- 只要原数还有数字没处理，就继续循环。

```csharp
int digit = x % 10;
```

- 取出 `x` 的最后一位。
- 例如：
  - `123 % 10 = 3`
  - `-123 % 10 = -3`

```csharp
x /= 10;
```

- 去掉最后一位。
- 例如：
  - `123 / 10 = 12`
  - `-123 / 10 = -12`

```csharp
if (result > int.MaxValue / 10 || (result == int.MaxValue / 10 && digit > 7))
{
    return 0;
}
```

- 这是正数溢出判断。
- 因为接下来要做：

```csharp
result = result * 10 + digit;
```

- 如果 `result` 已经大于 `214748364`，那乘以 `10` 一定爆。
- 如果 `result` 刚好等于 `214748364`，那最后一位最多只能再拼 `7`。
- 因为 `int.MaxValue = 2147483647`。

```csharp
if (result < int.MinValue / 10 || (result == int.MinValue / 10 && digit < -8))
{
    return 0;
}
```

- 这是负数溢出判断。
- 如果 `result` 已经小于 `-214748364`，再乘 `10` 一定越界。
- 如果 `result` 刚好等于 `-214748364`，最后一位最小只能拼 `-8`。
- 因为 `int.MinValue = -2147483648`。

```csharp
result = result * 10 + digit;
```

- 把刚取出来的最后一位，拼到反转结果后面。

## 为什么判断里是 7 和 -8

- `int.MaxValue = 2147483647`
  - 最后一位是 `7`
- `int.MinValue = -2147483648`
  - 最后一位是 `-8`

所以边界判断里才会出现：

- `digit > 7`
- `digit < -8`

## 用例走一遍

### `x = 123`

- 第 1 轮：
  - `digit = 3`
  - `x = 12`
  - `result = 3`
- 第 2 轮：
  - `digit = 2`
  - `x = 1`
  - `result = 32`
- 第 3 轮：
  - `digit = 1`
  - `x = 0`
  - `result = 321`

返回 `321`

### `x = -123`

- 第 1 轮：
  - `digit = -3`
  - `x = -12`
  - `result = -3`
- 第 2 轮：
  - `digit = -2`
  - `x = -1`
  - `result = -32`
- 第 3 轮：
  - `digit = -1`
  - `x = 0`
  - `result = -321`

返回 `-321`

## 为什么这是最优解

- 不需要字符串转换。
- 不需要 `StringBuilder`。
- 不需要 `Reverse()`。
- 不需要 `TryParse()`。
- 不依赖 64 位整数。
- 空间复杂度是 `O(1)`。

## 复杂度

- 时间复杂度：`O(log10 n)`
- 空间复杂度：`O(1)`

## 复盘

- 以后看到“整数的每一位”这种题，优先想：
  - `% 10` 取最后一位
  - `/ 10` 去最后一位
- 如果题目明确提到 32 位范围、不能用 64 位，通常就是在提醒我要自己做边界判断。
- `TryParse` 可以作为字符串法的兜底，但不是这题最核心的考点。

## 一句话记忆

- 先取尾，再缩短，再拼回去；拼之前先判断会不会炸。
